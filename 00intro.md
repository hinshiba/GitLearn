## Gitを使おう

### なぜGitを用いるか

Gitは，ファイルの変更履歴を記録し，管理するためのツールである．このような仕組みを一般にバージョン管理システムと呼ぶ．

バージョン管理を行わないと，次のような問題に直面する．

- いつ，誰が，どこを変更したのかがわからなくなる
- 過去の状態に戻したくても戻せない
- 複数人で同じファイルを編集すると，変更が上書きされて消えてしまう

例えば，レポートやプログラムを書く際に，`report_final.docx`，`report_final2.docx`，`report_本当に最終.docx`のようにファイルを複製して管理した経験はないだろうか．これは原始的なバージョン管理だが，どれが最新かわからなくなったり，他人の変更と統合できなかったりと，すぐに破綻する．

Gitを使うと，次のことができるようになる．

- 変更の履歴を意味のある単位(コミット)で記録し，後から振り返れる
- 任意の過去の時点の状態を復元できる
- 複数人がそれぞれ作業した変更を，安全に統合できる
- リモートリポジトリと連携することで，変更の共有とバックアップを同時に行える

特にチーム開発では，Gitは事実上の標準であり，使えることが前提とされる場面が多い．

### 大まかな全体の流れ

個々のコマンドの詳細は後のレッスンで扱う．ここでは，開発の中でGitをどのような順序で使うのか，全体像をつかんでほしい．

#### プロジェクトを新しく作る人

手元のディレクトリをGitの管理下に置く(リポジトリの作成)．

```bash
git init
```

リモートリポジトリを設定する．

```bash
git remote add origin <リモートリポジトリのURL>
```

#### 既存のプロジェクトに参加する人

リモートリポジトリを手元に複製する．

```bash
git clone <リモートリポジトリのURL>
```

これにより，リモートリポジトリは自動的に設定される．

#### 作業ごと

最新の状態を取り込み，作業用のブランチを作成する．

```bash
git pull
git switch -c feat/hoge
```

ファイルを編集し，区切りのよい単位でコミットする．

```bash
git add .
git commit -m "feat: hogeの追加"
```

編集とコミットを，機能が完成するまで繰り返す．

完成したら，変更をリモートリポジトリへ反映する．

```bash
git push
```

最後に，自分の変更を本流に取り込んでもらうため，プルリクエスト(マージリクエスト)を作成する．
チーム開発ではレビューを受け，承認されると変更が統合される．

個人開発ではAIによるレビューや，mainへの直接プッシュ等が行われる．

### gitの設定

Gitを使い始める前に，最低限の設定を済ませておく．設定を整えておくことで，環境による予期しない挙動や，毎回の余計な操作を避けられる．

まず，コミットに記録される名前とメールアドレスを設定する．これは必須であり，設定しないとコミットできない．

```bash
git config --global user.name "github等に登録したユーザー名"
git config --global user.email "github等に登録したメールアドレス"
```

次に，以下を設定ファイル（`~/.gitconfig`）に追記しておくとよい．

```conf
[init]
	defaultBranch = main

[push]
	autoSetupRemote = true

[fetch]
	prune = true
```

それぞれの意味は次のとおりである．

- `init.defaultBranch = main`
  `git init`で作られる最初のブランチ名を`main`にする．多くのホスティングサービスが`main`を既定としているため，これに合わせる．
- `push.autoSetupRemote = true`
  新しいブランチを初めてpushするとき，対応するリモートブランチを自動的に設定する．これにより，毎回`git push -u origin ブランチ名`と指定せず，`git push`だけで済むようになる．
- `fetch.prune = true`
  リモートで削除されたブランチに対応する追跡ブランチを，`fetch`時に自動で整理する．手元に不要な参照が残るのを防げる．
