## ローカルでの復元

コミットを変更の保存と説明したように，コミットした状態まで巻き戻す方法が存在する．
それぞれに特徴があるため，適切に使えるようになってほしい．

登場するコマンド
- `git restore`: まだコミットしていない変更を捨てたい / ステージングエリアの変更を消したい
- `git revert`: コミットを安全に取り消したい
- `git reset`: まだプッシュしていないコミットを履歴を残さず消したい

大まかには，`git restore`はまだコミットしていない変更を戻すもの，`git revert`と`git reset`はコミット済みの変更を戻すものである．

### コミットの状態に戻す

ワーキングツリーの現在の変更を破棄し，現在のステージングエリアの状態に戻すには`git restore`を用いる．

```bash
git restore <ファイル名>
git restore --worktree <ファイル名> # 同じ
```

`git add`と同様に，`.`を指定することで，すべてのファイルを対象とできる．
ステージングエリアの状態に戻すため，`git add`によってステージングされたファイルは復元されない．

ステージングエリアを現在のコミットから復元する操作も同じコマンドである．

```bash
git restore --staged <ファイル名>
```

この場合，ワーキングツリーの状態は変更されない．

両方を変更するためには，両方のオプションを付与したコマンドを用いる．

```bash
git restore --staged --worktree <ファイル名>
```

VSCodeでは，ソース管理ビューで変更されたファイルにカーソルを合わせると現れる「変更を破棄」(矢印が回転したアイコン)から，コミット前の変更を破棄できる．
また，マイナスアイコン`-`の「変更のステージング解除」を用いることで，ステージングエリアをコミットの状態に戻すことも可能である．


### 特定のコミットを取り消す

ときにはコミットの後に誤りに気付くこともある．
`git revert`は，指定したコミットの変更を打ち消す新しいコミットを作成するコマンドである．

```bash
git revert <コミット>
```

過去のコミットを消去しないため，後の章で説明するリモートリポジトリにプッシュした後でも安全に使用できる．

### 特定のコミットまで巻き戻す

`git reset`は，指定したコミットまで各種プロジェクトの状態を巻き戻すコマンドである．
移動先より後のコミットは基本的には消滅する．

```bash
git reset <コミット>
git reset --mixed <コミット> # 同じ
```

どの状態を戻すかをオプションで指定できる．

| オプション | コミット履歴 | ステージングエリア | ワーキングツリー |
| ---------- | ------------ | ------------------ | ---------------- |
| `--soft`   | 戻す         | そのまま           | そのまま         |
| `--mixed`  | 戻す         | 戻す               | そのまま         |
| `--hard`   | 戻す         | 戻す               | 戻す             |

どのような操作においても，コミット履歴が変化するためすでにリモートにプッシュしている場合は非推奨である．

> `git reset --hard c1`は`git reset --soft c1`の後に`git restore --staged --worktree .`を行うことと同等である．
